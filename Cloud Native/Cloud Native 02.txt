
Cloud Native 2

■ Cloud Native 도입 
     - 도구나 설계의 영향력과 한계를 이해하고 도입해야 함 (맹목적으로는 안됨)
     - 질문 항목
          + 클라우드 네이티브 애플리케이션을 사용하는가?
          + 엔지니어링 팀이 관리할 수 있는가?
          + On-premise나 IaaS 가 있는가? (적어도 계획이라도)
          + 더 빠른 개발주기와 비선형적 확장이 필요한가?
      - 처음 잘못 도입하면 오히려 좋지 않다는 편견이 생김  

==> Cloud Native Infra의 고려사항


■ Application
     - 상대적으로 준비가 쉬움
     - 빌드/검증/테스트된 파이프라인을 통해 자동으로 배포가 안 되는 경우 서둘러서는 안됨
     - 반드시 MSA 구조를 의미하지도 않음 
     - 애플리케이션 인스턴스의 동적 확장이 가능 여부 (사람의 개입 없이…)

■ Human
     - 가장 난제, 사람은 확장되지 않으며, 반복적이며 일상적인 업무에 능숙하지 않다. 
     - 비즈니스 논리와 의사결정을 포함할 수 있는 SW 개발 기술이 필요
     - 팀 구성과 의사소통 구조가 중요

■ System
     - 시스템의 추상화가 필요 (하드 코딩 X)
     - 신뢰성 
     - On-prem/Hybrid/Outsource ???

■ Enterprise
    - 기업이 만들어야 하는 인프라와 애플리케이션에 의사소통 구조가 반영되어야 함
    - 애자일한 애플리케이션의 증가는 제한요소
    - 반복과 변경을 더 자주 실행하는 것을 인지해야 함

■ Cloud Native Infra가 필요 없는 상황
    - 기술적 한계 
         + 클라우드 네이티브 애플리케이션이 없는 경우
         + 자동화가 부족한 경우
         + 신뢰성 문제
    - 기업의 한계 
         + 정책이나 절차가 부족

         + 호스팅 서비스 도입에 소극적이거나 의사가 없는 경우
